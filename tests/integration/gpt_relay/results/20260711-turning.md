## A/B conclusion

The same relay/model works under both provider names with API-key auth. Setting
the display name to `OpenAI` materially changes the second-turn wire body by
adding sequential reasoning-summary delivery and retaining internal item
metadata, and TURNING accepted those differences.

The compatibility break requires the additional Codex-backend-auth condition:
that combination enables Zstd and TURNING returned HTTP 400. The non-OpenAI
backend-auth control succeeded, so the failure is attributable to the
OpenAI-only request shape rather than the model or base route. A middle layer
can preserve model-switch compact fallback if it decodes Zstd before forwarding.

The Gateway subsequently implemented this bounded inbound decoding path. The
same live WebUI `request_body_limit_mb` value is enforced once on compressed
wire bytes and again on decompressed bytes; uncompressed requests do not enter
the decoder.

A post-implementation isolated run completed successfully through the actual
Gateway at `tmp/agent_testing_workspace/202607111932`. The bounded Gateway trace
at `/Volumes/RAMDisk/202607111932/rosetta-trace.jsonl` records
`provider_name=TURNING`, source/target `openai_responses`, and actual model
`gpt-5.6-terra`; the Codex source harness completed the OpenAI-identity Zstd
turn without the earlier HTTP 400.
