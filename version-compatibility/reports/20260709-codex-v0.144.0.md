## 执行阶段

### 阶段 1：补强源码变化检测

1. 扩展 extractor，捕获完整 web-search tool 字段、`AdditionalTools` wire shape、
   `ModelMessages.approvals` 和 bundled model 的关键能力值。
2. 为 tag commit、同步后的 source commit 和 package release version 保留独立字段。
3. 新增 extractor fixture，确保字段 rename、嵌套 schema 和 catalog 值变化进入“有变化”。

### 阶段 2：修复阻断兼容点

1. 先实现 Responses Lite request 解析：恢复工具、developer instructions、history 和
   reasoning context；定义顶层/嵌入工具混合时的去重与优先级。
2. 让 `remove_image_generation` 同时识别旧 `image_generation` 和新 `image_gen`
   namespace/function，并处理 `additional_tools` 内嵌定义。
3. 更新 web-search 新字段和 hosted/extension 两条路径；补 `bio_policy` error contract。
4. 增加 `ultra` reasoning 与 provider-specific 降级；保持 encrypted/summary 状态续轮。

### 阶段 3：收紧历史、deferred tools 与 stream

1. 对 bridge 的 item ID、call ID、added/done/completed 关联增加确定性回归。
2. 覆盖 canonical/extension TurnItem、MCP snapshot、image namespace、compact/resume/
   fork/restart 和两个 window 的状态隔离。
3. 只有真实捕获显示 header 生命周期变化时，才把 window key 从 header 扩展到 body
   `client_metadata`；不基于猜测改状态所有权。

### 阶段 4：运行全部自动化门禁

1. 运行 Codex 专项 pytest，包含本报告新增的 Lite/catalog/web-search/error/history
   fixtures。
2. 运行 `make lint` 和 `make test`。
3. 在语义审查和修复完成前，`make check-codex-compat` 应继续失败；不得提前刷新
   baseline。

### 阶段 5：真实 Codex/API 测试

1. 默认使用 `deepseek-v4-flash`；对 Lite/code-mode 使用受控 model catalog override，
   确保真实触发目标 wire shape，而不是只测试 unknown-model fallback。
2. 合并执行文本/工具/Lite、patch 错误恢复、deferred plugin/MCP、web search、phase、
   reasoning、compact/resume/fork 和并发 window 场景，逐项回填上表结果。
3. `bio_policy` 不主动用敏感真实 prompt 触发，使用 fixture；真实错误门禁改用安全的
   无效参数/模型路径并记录替代原因。

### 阶段 6：收尾与版本升级

所有修复、自动化和触发的真实测试通过后，更新 contract baseline、本报告、当前
兼容基线和版本，并把 Codex-Rosetta 包版本从 `0.142.5` 提升到 `0.144.0.r0`。
baseline 记录实际审查的 `2e8c375…`，同时保留 release tag commit `7678224…`。
